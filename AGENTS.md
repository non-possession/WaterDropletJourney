# AGENTS.md

## 《水的旅程》修改时必须牢记的核心思想

### 1. 先保体验，再谈功能

- 这是一个“弱目标、强感受”的 2D 诗性叙事探索游戏。
- 每次修改都要优先判断：它是否让玩家更能感受到流动、关系、变化与归去。
- 不能把项目做成硬核平台跳跃、传统解谜堆砌或纯文字叙事。

### 2. 玩家是在经历，不是在阅读

- 叙事要尽量通过移动、停驻、附着、渗透、凝结、蒸腾、汇流来表达。
- 主题不能主要靠说明文字或对白承担。
- 如果一个改动只能“解释主题”，却不能“让玩家经历主题”，优先级就不够高。

### 3. 每个改动都要服务核心交互语言

本项目的核心交互语言是：

- 流
- 停
- 聚
- 散
- 变
- 依附
- 渗透
- 离开
- 汇入

任何新增机制、场景或反馈，最好都能落回这些关键词，而不是偏向战斗、征服或高压挑战。

### 4. 每章都必须同时成立“情绪目的”和“操作目的”

- 不能只有氛围没有可玩的体验。
- 也不能只有机制练习没有情绪重量。
- 每次修改章节内容时，都要检查这次改动是在强化情绪目的、操作目的，还是最好同时强化两者。

### 5. 世界要改变水，水也要留下微小痕迹

- 环境不是背景板，而是会回应水的对象。
- 玩家的动作必须和世界发生关系。
- 优先设计“被世界改变”和“留下痕迹”的互动，而不是“征服障碍”的互动。

### 6. 引导要柔和，不要把玩家从沉浸里拽出来

- 主路径需要清晰，但不能过度任务列表化。
- 引导优先使用光、声音、构图、材质、动态反馈。
- 少做生硬弹窗、强提示和密集 HUD 目标。

### 7. 水滴要有存在感，但保留投射空间

- 水滴不是普通角色控制器，也不是过度卡通的拟人角色。
- 表达优先依靠形变、亮度、运动节奏、停顿和声音质地。
- 不要让过多台词和强设定压死玩家的情感投射。

### 8. 文本永远排在体验之后

- 信息传达优先级应是：体验 > 环境 > 事件 > 文本。
- 文本要少、准、留白，并且最好一条承担多重作用。
- 任何一句诗性表达都最好能对应到具体可感的玩法或环境状态。

### 9. 视听表达不是装饰，是体验主体

- 视觉要服务柔和、写意、可感知的状态变化。
- 声音是核心表达层，不是陪衬。
- 每次修改镜头、颜色、UI、音效时，都要检查它是否在强化“陪它走过一段”的感受。

### 10. 一致性比复杂性更重要

- 规则、手感、环境反馈、情绪表达必须彼此支持。
- 不要为了丰富而堆太多机制、分支或例外。
- 优先做清晰、统一、可感知的状态变化和环境关系。

### 11. 非线性追求“个人经历感”，不是分支数量

- 不追求庞大剧情树。
- 更适合通过探索顺序、环境碎片、不同形态下的空间意义变化来制造“这是我经历的旅程”。
- 避免高成本、低意义的伪选择。

### 12. 低压操作本身就是审美策略

- 不用高失败惩罚破坏情绪连续性。
- 不让高操作压力成为理解作品的门槛。
- 修改手感、难度、碰撞和节奏时，要优先保护细腻感和陪伴感。

### 13. 大目标必须绑定任务简报

- 每一次大的项目目标，都必须有一份对应的任务简报，优先放在 `docs/tasks/` 中。
- 执行目标过程中的小变更，不能只留在聊天或 commit message 里，要作为增量记录写回对应任务简报。
- 任务简报必须包含明确验收属性、验收证据、当前进度、下一步风险和边界条件。
- 每条计划或变更都要记录时间戳和对应 Git commit 号；如果还没有 commit，要先标记为“未提交”。
- 任务简报要维护“验收属性覆盖率指数”，用 `已有可验证证据的属性数 / 总属性数` 表示，并在关键变更后更新。
- `docs/` 仍然是私有项目管理区，默认不要提交到公开仓库；若需要公开某条规则，应同步写入 `AGENTS.md` 或其他可公开文档。

## 每次修改前的快速自检

- 这个改动是否让玩家更能“感到”，而不只是“知道”？
- 这个改动是否服务于流动、关系、变化与归去？
- 这个改动是否强化了某章的情绪目的和操作目的？
- 这个改动是否让环境更像叙事参与者，而不是背景？
- 这个改动是否保持了低压、柔和、可沉浸的体验契约？
- 如果这是大目标或关键增量：是否更新了对应任务简报、时间戳、commit 号和验收属性覆盖率指数？

如果答案多数是否定的，就需要重新考虑实现方向。
